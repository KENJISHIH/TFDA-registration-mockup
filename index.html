#1

全成分登錄優化

資料字典化 + 批次匯入 + 法定警語自動帶入（A 章主戰場）

全成分登錄是化粧品登錄系統最複雜、最容易出錯、影響最深遠的環節。 本提案分三大改革主題：資料品質、操作流程、法定警語連動，三者環環相扣。

BEFORE — 現行 3 大痛點

⚠ 痛點 A：資料分裂

CAS 7732-18-5 · 7 筆同一物質：
WATER ／ AQUA ／ AQUA / WATER ／
WATER (AQUA) ／ WATER / AQUA ／ ...
  • −同 INCI／同 CAS 散成多筆
  • −萃取物無 CAS，主鍵設計失靈
  • −業者自由輸入文字，無 canonical reference

⚠ 痛點 B：操作流程

  • −雖提供 Excel 模板匯入，但須離線編輯後再上傳，無即時驗證
  • −搜尋僅支援前綴比對（輸入「glycer」才找到 GLYCERIN）
  • −常用配方無法存範本，每次重新輸入
  • −無 API 介接，業者 ERP 無法直接 push 與雙向同步

⚠ 痛點 C：警語遺漏

  • −限制成分對應法定警語靠業者記憶
  • −TFDA 公告修訂後，業者後台無自動更新
  • −AI 合規工具無法事前預檢

AFTER — 3 大改革主題

擴充 B

成分變更分級登錄（minor / major）

現行：成分一動就視為「新案重新登錄」，業者重繳費、重走流程。建議以「是否影響原始安全評估與警語結論」為分級判定基準。

Minor change · 變更登錄 基本費

香料配方微調、移除非限制成分、添加量低於閾值（不影響安全評估）

Major change · 重新登錄 新案費

新增／移除限制成分、改變產品劑型、影響安全評估或警語

📐 分級判定原則

分級依據是「是否影響原始安全評估與使用注意事項」： PIF 安全評估與警語結論不變 → minor；需重做安全評估或新增/刪除警語 → major。 歐盟 CPNP（Reg. 1223/2009 Art. 13）採類似精神：當 frame formula 不再對應實際配方、成分濃度顯著變化、加減有害成分時，業者應主動更新通知。

  • +登錄辦法第 5 條「變更登錄」現行已存在，本建議僅擴充其適用範圍至成分微變更
  • +須由 TFDA 公告「變更分級判定基準」（成分清單 + 含量閾值）
  • +對應 D-5 事件驅動：minor／major 觸發不同事件流

法源： 登錄辦法第 4 條（應登錄事項）／第 5 條（登錄事項變更者，應辦理變更登錄）。
架構支撐：對應本提案 D 章 D-1 永久 ID（永久識別碼於 submitted 時刻發出）／ D-2 欄位 Schema（必填／選填屬性 schema 化）／ D-5 事件驅動（draft→submitted、minor／major 變更為不同事件）。

#7

品名風險預警與法規透明化

把既有公開研析成果落地為業者主動避險工具——從「事後處罰」轉為「事前協助」

提案立場：透明化、不增加限制

研析背景：業界已有公開可參考之 37 項風險詞句研析（非屬化粧品 11 項／屬性疑慮 6 項／易混淆詞 20 項）。

現況落差：研究成果雖已公開，但多數業者並不知此清單存在，導致登錄時誤踩風險。

本案訴求：讓研究成果在登錄系統中「主動觸達業者」——不是新增限制，而是讓既有資訊透明化、可預警。

即時輸入預警（非強制）

頭皮新生活化精華液 ⚠ 含風險詞句
偵測到「新生」「活化」—— 依現行公開研析歸為「易混淆詞」（毛髮／頭皮類）， 可能讓消費者聯想醫療效能。

💡 系統建議：

可考慮替代用語（例：「頭皮舒養精華液」），或維持原品名繼續登錄——業者保有選擇權，風險自負。

三級風險分級提醒

Level 1 · 非屬化粧品

這類用語可能讓產品被認定不屬化粧品範疇

例：廁後點滴、口腔噴劑、防蚊用品

Level 2 · 屬性疑慮

這類用語可能引起對產品屬性的疑問

例：舒緩貼布、銀離子噴霧

Level 3 · 易混淆詞

這類用語可能讓消費者聯想醫療效能

例：防落髮、青春永駐、術後修復

設計原則

  • →系統定位為「業者預警工具」，不是強制限制；偵測到風險詞句僅顯示提醒與建議
  • →業者可選擇調整品名或繼續使用，最終決定權在業者；行政裁量權保留於主管機關承辦
  • →核心價值：透明化既有研究成果，降低業者誤觸風險，從「事後處罰」轉為「事前協助」

研析基礎：現行公開之化粧品標示宣傳廣告涉及不適當字詞研析（37 項三類）。
架構支撐：對應本提案 D 章 D-2 欄位 Schema（風險詞句結構化資料字典）／ D-5 事件驅動（研析清單更新時推播業者）／ D-8 OpenAPI（業者 ERP 可離線預檢，無需登入 portal）。

  • −人員資料 vs 產品登錄系統各自獨立

✓ 主題 A：人員自主登錄平臺

#16

代工廠「不實掛名」異議通報機制

為現行「登錄製造場所反查」功能補上後續處理管道——資訊透明補強，不變動法規責任歸屬

立場聲明： 化粧品衛生安全管理法明定登錄義務人為品牌商（製造業者），代工廠並非登錄主體； 本案不變動現行法規之責任歸屬。 登錄系統已建置之「登錄製造場所反查」功能值得肯定 —— 代工廠已可查詢自身名下之產品清單； 然而「看到不實掛名後沒有正式通報管道」之缺口仍待補上。 本案僅就此面向提供輔助機制，所有行政裁量權完全保留於主管機關承辦。

BEFORE — 反查可見，但「看到」不能「處理」

✓ 現況肯定

登錄系統已建置「登錄製造場所反查」功能，代工廠可查詢自身名下之產品清單。

實際痛點：看得到、做不到

  • −已停止代工，產品仍在我廠名下
  • −部分產品根本未在我廠生產，仍被掛名
  • −代工廠看到不實掛名，無正式通報／更正管道
  • −只能私下與品牌商溝通，溝通失敗即無解

AFTER — 反查功能 + 異議通報管道

① 反查功能升級為主動通知（選擇性）

代工廠可選擇開啟自動通知 —— 品牌商登錄／變更登錄時系統自動 email／站內訊息告知，取代「定期主動登入查詢」之負擔。

② 異議標註（核心新增）

代工廠對反查清單中之產品可逐筆標註：

✓ 正常代工預設值，無需操作
⚠ 已停止代工過去有但現在沒有
✗ 實際未代工不實掛名

③ 異議後處理流程

  1. 1.系統自動通知品牌商，請其於期限內確認或更正
  2. 2.品牌商如更正登錄資料，案件自動結案
  3. 3.品牌商未回應或回應有爭議 → 由主管機關依職權查核

代工廠之標註僅為輔助參考，不直接影響登錄效力，最終裁量權保留於主管機關

  • +補上現行反查功能之「處理管道」缺口
  • +取代不可行之「主動檢舉」要求

D-1

永久不變的資源 ID

產品登錄號碼、業者 ID、場所 ID 一旦發出就不重用、不變動

為什麼重要： ID 是所有外部引用的錨點——條碼、QR Code、ERP、海關、電商上架、消費者 APP 都靠它對齊。 ID 一旦變動，所有引用立即失效，且失效是「靜默」的（消費者掃出空白、海關抓不到對應產品），事後補救成本極高。

BEFORE — 現行做法的隱憂

同一支產品的 ID 演變

2024 FDA-A-2024-001234 初次登錄
2025 FDA-A-2024-001234 業者改名 → ID 重發
2025 FDA-B-2025-007821 新 ID（與舊 ID 無連結）
2026 FDA-B-2025-007821 展延 → ID 又換
  • −業者異動、展延、變更登錄都可能觸發新 ID
  • −包裝印刷的條碼、電商商品頁失效卻無人察覺
  • −稽查、回收追溯需人工串接歷史 ID，極易遺漏
  • −未來任何 API 對接都要先解決「同產品多 ID」問題

AFTER — 永久 ID 架構（建議）

同一個 ID，跨年跨系統穩定引用

  • +登錄號碼一旦核發即為永久 ID，業者改名、地址、展延等「變更登錄」皆不換號
  • +屬性變更以「版本記錄」附掛在同一 ID 下，可追溯歷史（成分變更之「重新登錄」例外，法律上已是新產品）
  • +條碼、QR、ERP、海關、消費者 APP 共用同一錨點
  • +未來開放任何 API，外部系統只需對接一個 ID 概念

法源相容性

本原則不涉及法規修正。化粧品產品登錄辦法第 4 條第 1 項第 1 款已明定「產品登錄號碼」為登錄資料； 同辦法第 7 條並區分二類法定行為—— 「變更登錄」（其他事項變更，如業者改名、地址、展延等）與 「重新登錄」（涉及成分變更，法律上已視為新產品）。 本原則僅要求「變更登錄」時不得異動識別碼；「重新登錄」因法律上已為新產品，發新識別碼合理。 識別碼之穩定範圍以「同一登錄案」為單位，屬規格層次決定，主管機關得於系統規格書中明訂。

給規格書的具體建議文字

「產品登錄號碼、業者識別碼、製造場所識別碼一經核發，即為永久識別碼，
不得重用，不得於『變更登錄』時異動。屬性變更（業者改名、地址、展延等）
以版本記錄附掛於同一識別碼之下；『重新登錄』（成分變更）
法律上已視為新產品，得發新識別碼。」

D-2

欄位 Schema 與資料字典

每個欄位有明確型別、值域、必填性、版本號

為什麼重要： 欄位定義是介接的「合約」。可填什麼、長度多少、是否必填、版本何時變——這些一旦只藏在前端 JavaScript 裡， 業者只能靠人工試錯找出規則，AI 工具無法做合規前置檢查，跨機關介接更只能反向解析網頁。 把欄位定義公開、版控、機器可讀，是後續所有 API、表單、稽核工具的共同基礎。

？ 最多幾個字／格式限制／特殊符號可不可以

  • −欄位驗證僅存於前端 JavaScript，業者只能反推
  • −選單值域未公開，無法做離線預檢
  • −規則修訂無版本紀錄，舊資料是否仍合規無從判斷

D-3

內部系統 Dogfooding

TFDA 內部稽查、查詢、報表都吃同一組 API

為什麼重要： 如果 TFDA 內部系統與未來對外 API 各自實作，兩邊規格永遠不同步、業務邏輯永遠重複。 讓內部稽查、查詢、報表「自家先吃自家的菜」，未來對外開放只是「授權打開既有 API」， 而不是重新打造一套——後者成本與風險都高出數倍。

BEFORE — 內外雙軌、規格分裂

內部直連 DB、外部沒有路徑

  • −內部系統直連 DB，業務邏輯散落各處
  • −未來對外開放需另寫一套 API，規格不同步
  • −同一份資料、兩套查詢，行為微妙不一致
  • −內部 API 直到開放當天才接受外部驗證

AFTER — 統一 API Gateway

內外部共用同一組 API 與業務邏輯

  • +內外部共用同一組 API，零重複實作
  • +內部使用即「自家先吃自家的菜」，bug 提早暴露
  • +對外開放只需「授權打開既有 API」
  • +規格、版本、欄位三者天然一致

規格定位

本原則屬系統架構選擇，不涉及法規修正，亦不變更登錄事項。 可援引國發會《政府資訊系統整體規劃》或行政院相關統籌精神，強調 API 共享、避免重複開發。

給規格書的具體建議文字

「TFDA 內部稽查、查詢、報表、統計等系統，應與未來對外開放之 API 共用同一介接規格與業務邏輯層。
內外部 API 不應分屬不同實作；對外開放屬授權層次之決定，非另行打造系統。」

D-4

「業者帳號介接應採 OAuth 2.0 與 OpenID Connect (OIDC) 標準協定。
業者得授權第三方系統（如 ERP）以 access token 代理操作，
授權範圍（scope）、效期（expiry）、撤銷機制應符合 RFC 6749 / RFC 8252 規範。」

附錄

密碼政策現代化：取消 90 天強制變更

現行 GSP 單一登入即便已採 OIDC，仍要求每 3 個月強制變更密碼。此政策已被國際資安標準淘汰—— 強制定期變更會導致使用者採用「弱化版本」（password1 → password2 → password3），反而降低安全性。 既然 GSP 走 OIDC，即應採用現代 IdP 政策，與 1990 年代密碼輪替思維脫鉤。

❌ 現行（已過時）

  • −每 90 天強制變更密碼
  • −使用者採弱化版本（Password01 → Password02）
  • −無 MFA、無外洩監控、無異常登入偵測
  • −業者操作體驗差，提高 reset 工單量

✓ 建議（NIST 800-63B 對齊）

  • +取消強制 90 天變更（除非有外洩證據）
  • +強制 MFA（TOTP／FIDO2／WebAuthn）
  • +密碼外洩監控（接 Have I Been Pwned API）
  • +異常登入偵測（地理位置、裝置指紋）
  • +弱密碼字典阻擋（最低 12 字元、字典過濾）

🌏 國際資安標準依據

  • NIST SP 800-63B（2017 修訂、2024 最新版）：明確不建議強制定期密碼變更，僅在外洩證據存在時要求重設
  • Microsoft Security Baseline（2019 起）：移除 60-day password expiration policy
  • NCSC（英國國家網路安全中心）／CISA（美國）：同樣立場——強制輪替反而降低安全
  • Google、AWS、Apple：全面取消強制密碼變更，改採 MFA + 風險導向認證

給規格書的具體建議文字

「使用者密碼政策應參照 NIST SP 800-63B：
取消強制定期變更要求；
改強制啟用多因素驗證（MFA）；
密碼長度下限 12 字元、阻擋常見弱密碼字典；
偵測外洩或異常登入時觸發強制重設。」

D-5

事件驅動架構

登錄狀態變更觸發 webhook／訊息佇列，毫秒級扇出

為什麼重要： 登錄狀態變更（廢止、停止、撤銷）影響極廣——關務署放行、電商上架、消費者購買決策都依賴此狀態。 現行作法是各方各自輪詢，延遲少則數小時、多則數日。事件驅動架構讓狀態變更瞬間扇出至所有訂閱者， 建立全國共享的「單一事實來源」。

BEFORE — 各方輪詢、延遲累積

T+6h 關務署：仍視為有效（次日輪詢）

T+18h 消費者 APP：顯示有效（無此功能）
T+24h 內部稽查：報表才出現變更
  • −關務署放行可能比 TFDA 廢止晚 24 小時
  • −電商繼續販售已下架產品
  • −新增訂閱者都要 TFDA 協助寫排程
  • −「狀態當前值」沒有單一事實來源

AFTER — Event Hub 毫秒級扇出

廢止事件 → 全部訂閱者，T+毫秒

  • +業者按下「廢止」毫秒級扇出至所有訂閱者
  • +新訂閱者自助接入，無需 TFDA 協助
  • +全國共享「狀態當前值」單一事實來源
  • +消費者保護瞬時生效，回收下架同步

規格定位

本原則屬技術架構選擇，不涉及法規修正。事件本身即現行已存在之登錄狀態變更，僅多新增「對外推播管道」， 不增加業者義務。

給規格書的具體建議文字

「登錄狀態（包含但不限於：核准、變更、廢止、撤銷、停止）變更時，
系統應發出標準事件訊息（建議採 CloudEvents 規範），
透過 webhook 或訊息佇列推播至已訂閱之內外部系統。」

D-7

PIF 訂為機器可讀格式

用足母法第 4 條第 3 項授權，把 PIF 從 Word 升級為結構化檔

為什麼重要： 產品資訊檔案 (PIF) 依《化粧品產品資訊檔案管理辦法》第 7、8 條，係**業者自家保存、主管機關派員查核**之備查制度，本原則完全不變動此前提。 但目前各業者 PIF 格式各異——業者自家版本管理混亂、AI 合規工具進不來、稽查員到場面對不同 Word 模板都要從零理解。 辦法第 7 條已規定 PIF 得以「**電子儲存**」方式保存，主管機關得在此授權內進一步訂定電子儲存之標準格式（如 JSON Schema）， 讓**業者自家、稽查現場、登錄系統**三方同時受惠。

BEFORE — Word 檔散落業者硬碟

現行備查與查核流程

PIF_產品A_v3_最新版.docx

業者自家保存（辦法第 7 條），各廠格式不一

↓

主管機關依第 8 條派員前往查核（前 7 日通知）

↓

每家格式不同，稽查員從零理解、人工翻找

  • −業者自家：版本管理混亂，AI 合規工具完全進不來
  • −稽查現場：每家業者 Word 模板各異，從零理解極耗時
  • −登錄系統與 PIF 各成孤島，產業趨勢分析無解
  • −業者錯失「自我查核紅利」（在被派員查核前自我體檢）

AFTER — 結構化 JSON Schema

備查制度不變，僅將「電子儲存方式」標準化

pif-{product-id}.json

標準 JSON Schema · 仍存於業者端（辦法第 7 條）

↓

業者自家：ERP／品管／AI 合規工具直接讀寫

↓

稽查到場：標準工具開啟 · 跨家業者一致體驗

登錄系統側查詢（非 PIF 直查）：
SELECT registrant, product_name FROM registration
  WHERE 'PHENOXYETHANOL' IN ingredients AND status = 'active';
→ 據此清單依辦法第 8 條排定派員查核 PIF

  • +業者面：自家 ERP、品管、AI 合規工具直接讀寫 PIF，被查核前可自我體檢
  • +稽查面：到場用標準工具開啟，跨家業者一致體驗，效率提升
  • +登錄系統面：從登錄資料瞬時匯整「含特定成分產品清單」，據此排定派員查核重點
  • +備查制度不變：業者保存、派員查核之原則完全保留

法源依據（雙層授權，備查制度不變）

本原則之法源由兩層構成，完全不變動「業者保存、派員查核」之備查制度：
① 化粧品衛生安全管理法第 4 條第 3 項——授權主管機關訂定 PIF 之「項目、內容、變更、建立與保存方式」。
②《化粧品產品資訊檔案管理辦法》第 7 條——已規定 PIF 得以「電子儲存方式」保存。
「電子儲存」即「方式」之具體型態，主管機關得於該授權內進一步訂定電子儲存之標準格式， 業者保存地點（第 8 條）、派員查核程序（第 8 條）、保存期限（第 7 條）皆維持不變。

給規格書的具體建議文字

「產品資訊檔案 (PIF) 仍依《化粧品產品資訊檔案管理辦法》第 7、8 條
由業者自家保存，主管機關派員查核之備查制度不變。
依母法第 4 條第 3 項及辦法第 7 條『電子儲存方式』之授權，
PIF 之電子儲存應採機器可讀標準格式（建議採 JSON Schema 結構化），
便利業者自家品管、AI 合規工具讀寫，並利於稽查員到場以標準工具瀏覽。
業者得自願將結構化 PIF 同步至登錄系統，但同步與否不影響備查制度之適用。」
